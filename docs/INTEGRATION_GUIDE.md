# VETO Integration Guide

## Who this is for

This guide targets developers integrating VETO into:
- ERP systems generating B2B or B2G invoices
- Billing platforms issuing European invoices
- PEPPOL access points performing pre-submission validation
- Accounting SaaS products requiring EN16931 compliance

---

## Pre-submission validation flow

The correct integration pattern:

```
Invoice generated by ERP / billing system
    │
    ▼
POST /v1/validate (VETO)
    │
    ├── decision: VALID → proceed to submission
    ├── decision: WARNING → review warnings, then submit or reject
    └── decision: INVALID → block submission, surface errors to user
```

**VETO is a pre-flight check, not a submission gateway.**  
You still submit the invoice via your PEPPOL access point or direct tax authority channel.

## Minimal integration example (TypeScript)

```typescript
const BASE_URL = 'https://api.veto.dev';
const API_KEY = process.env.VETO_API_KEY!;

interface Invoice {
  invoice_number: string;
  issue_date: string;
  seller: { name: string; country: string };
  buyer: { name: string; country: string };
  currency: string;
  total_amount: number;
  [key: string]: unknown;
}

async function validateBeforeSubmit(invoice: Invoice, profile: string) {
  const res = await fetch(`${BASE_URL}/v1/validate`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-API-Key': API_KEY,
    },
    body: JSON.stringify({ invoice, target_profile: profile }),
  });

  if (!res.ok) throw new Error(`VETO API error: ${res.status}`);
  const result = await res.json();

  if (result.decision === 'INVALID') {
    const errors = result.blocking_errors.map(e =>
      `${e.rule_id}: ${e.message} — ${e.remediation_hint?.description}`
    );
    throw new Error(`Invoice validation failed:\n${errors.join('\n')}`);
  }

  return result;
}
```

## ERP integration pattern

```
Before generating the EDI/XML document:
1. Build invoice object from ERP data
2. Call VETO /v1/validate
3. If INVALID: surface errors in ERP UI, block export
4. If WARNING: show warnings in UI, allow override with acknowledgment
5. If VALID: proceed to UBL/CII document generation and submission
```

Store the `request_id` from VETO's response alongside your invoice record for debugging. The `request_id` is not sensitive — it contains no invoice data.

## Billing platform integration

For SaaS billing platforms generating invoices in volume:

```typescript
// Validate in invoice creation pipeline
async function createInvoice(data: InvoiceData) {
  // 1. Validate before persisting
  const validation = await validateBeforeSubmit(data, 'EN16931');
  if (validation.decision === 'INVALID') {
    return { success: false, errors: validation.blocking_errors };
  }

  // 2. Persist invoice record
  const invoice = await db.invoices.create(data);

  // 3. Submit to PEPPOL / tax authority
  await submitToAccessPoint(invoice);

  return { success: true, invoice };
}
```

## PEPPOL access point integration

For access points validating before routing:

```bash
# Validate against PEPPOL BIS Billing 3.0 profile
curl -X POST https://api.veto.dev/v1/validate \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $VETO_API_KEY" \
  -d '{
    "invoice": {
      "invoice_number": "INV-2024-001",
      "issue_date": "2024-11-15",
      "seller": { "name": "NL Supplier", "country": "NL" },
      "buyer": { "name": "DE Receiver", "country": "DE" },
      "currency": "EUR",
      "total_amount": 500.00
    },
    "target_profile": "PEPPOL-BIS-3.0"
  }'
```

## Detecting placeholder rule packs

Until VETO implements full schematron validation, `rule_pack_hash` will contain `PLACEHOLDER`. Your integration should detect this:

```typescript
if (result.rule_pack_hash.includes('PLACEHOLDER')) {
  console.warn('VETO: using placeholder rules — not full schematron validation');
}
```

## Error handling

| HTTP Status | Action |
|-------------|--------|
| 200 + VALID | Proceed |
| 200 + WARNING | Surface warnings, allow override |
| 200 + INVALID | Block submission, show `blocking_errors` to user |
| 400 | Fix request schema — check `details` field |
| 401 | Fix API key |
| 413 | Invoice payload too large (>1MB) — check for accidental encoding |
| 415 | Set Content-Type: application/json |
| 5xx | Retry with exponential backoff; alert on sustained failure |
